Token Scanners Only See Half the Risk. Here's What They Miss.

📅 April 1, 2026 ⏱ 10 min read 🔧 integrity.molt v2.1 🧪 Real program scan: Nosana (NOS)
TL;DR Rugcheck, RugSniffer, and most Solana token scanners check what the token looks like — mint authority, holder concentration, liquidity. They don't check what the code does. The largest Solana exploits (Wormhole $320M, Mango Markets $116M, Slope $8M) were all code vulnerabilities, not metadata flags. integrity.molt v2.1 adds Phase 0: static code analysis — cargo-audit CVE scanning, cargo clippy lint, and 13 Solana-specific grep patterns — before the AI layer runs. We tested this on the Nosana (NOS) program and found 3 CVEs and 84 pattern hits, which the AI correctly triage to 0 confirmed runtime exploits. That's the difference between a scanner and an auditor.

The Problem with "Token Safety" Scanners

Open Rugcheck on any token and you'll see a familiar set of checks: Mint Authority Disabled, Freeze Authority Disabled, Top 10 Holders %, LP Locked. These are all token-level metadata signals — facts stored in the mint account and associated token accounts on-chain.

They're useful. A token with mint authority still enabled can have its supply inflated at will. One with 80% concentrated supply can be dumped. These are real rug vectors. But they describe the token's current state, not the behavior of the program that controls it.

⚠ The gap Rugcheck cannot close The three most costly Solana exploits were code vulnerabilities invisible to any token metadata scanner:

• Wormhole bridge ($320M, Feb 2022) — a missing signature verification check in the Solana program. The attacker minted 120,000 wETH with no collateral.
• Mango Markets ($116M, Oct 2022) — oracle price manipulation via an economic exploit in the lending program logic.
• Slope wallet ($8M, Aug 2022) — private keys logged to a centralized server, a code quality/security failure invisible on-chain.

Rugcheck would have given all three protocols a clean report. Mint authority: disabled. Top holders: distributed. LP locked. No flags. Meanwhile, the code was exploitable.

Phase 0 in Detail

0A — cargo-audit: CVE Scanning

cargo-audit checks every Rust dependency in the program's Cargo.lock against the RustSec Advisory Database — the Rust ecosystem's equivalent of CVE. It runs as:

cargo audit --json 2>/dev/null | tee raw/cargo-audit.json

The output is structured JSON listing each advisory: ID, severity, affected crate, patched versions, and whether a fix is available. This feeds directly into the Analyst Agent's context window.

0B — cargo clippy: Code Quality Lint

cargo clippy --message-format=json runs the full Rust linter suite. Security-relevant lints include integer overflow warnings, unused mutable references that indicate incorrect ownership, and panic-unsafe code paths. For Solana programs, panics in production are a DoS vector.

0C — Solana Grep: 13 Vulnerability Patterns

We maintain a manually-curated set of 13 grep patterns targeting the most common Solana program vulnerability classes, derived from the Sealevel Attacks repository and post-mortems from known exploits:

missing_signer_check
Account marked signer=false where authority required. Classic privilege escalation.
missing_owner_check
Account owner not validated against expected program ID. Allows fake account injection.
unchecked_arithmetic
Integer ops without checked_add/checked_mul or saturating_*. Overflow/underflow risk.
arbitrary_cpi
Cross-program invocation to user-supplied program_id. Can call malicious programs.
missing_bump_canonicalization
PDA derived without canonical bump seed. Allows creation of multiple valid PDAs.
sysvar_clock_deprecated
Uses deprecated Clock sysvar access pattern instead of Clock::get(). Indicates stale code.
account_realloc_no_zero
realloc() called without zero_init=true. Exposes uninitialized memory to attacker.
duplicate_mutable_accounts
Two accounts in instruction can refer to same pubkey. Aliasing vulnerability.
insecure_initialization
Init instruction reachable after first call. Re-initialization attack vector.
unchecked_token_program
Token program account not validated as spl-token or token-2022. Fake token program risk.
invoke_signed_no_check
invoke_signed() without validating that seeds match the account being signed for.
lamport_arithmetic_overflow
Lamport subtraction without underflow guard. Can steal native SOL from accounts.
closeable_not_zeroed
Account close without zeroing data. Revive-after-close exploit possible.

The 3 CVEs — and Why They're Not Exploitable On-Chain

All three CVEs were found in the build toolchain dependency graph, not in code that executes on the Solana runtime:

Crate Advisory Severity Context AI verdict
proc-macro2 RUSTSEC-2024-0370 medium Build dependency only — not compiled into deployed binary not exploitable
curve25519-dalek RUSTSEC-2024-0344 high Timing side-channel in scalar multiplication. On-chain execution model prevents timing attacks. not exploitable
openssl-sys RUSTSEC-2023-0044 medium CLI tooling dependency. Not present in on-chain program artifact. not exploitable

The curve25519-dalek RUSTSEC-2024-0344 advisory is the most interesting. It's a real high-severity timing side-channel vulnerability — on a normal server, an attacker making millions of API calls could potentially extract private key material. But Solana's deterministic BPF execution model makes timing attacks physically impossible: the runtime doesn't expose wall-clock time to programs, and execution is isolated per transaction.

Without the AI triage layer, a naive scanner would report "1 HIGH severity CVE" and leave the user to figure out the rest. With the Analyst Agent, the report reads: "curve25519-dalek RUSTSEC-2024-0344 is not exploitable in the Solana execution context. The vulnerability requires timing measurement of cryptographic operations, which BPF's deterministic execution model prevents. No action required."

Conclusion

The Wormhole exploit didn't show up as a missing Mint Authority Disabled flag. It showed up in a .rs file as a missing signature check. Slope's $8M loss didn't have a "LP Unlocked" warning — it had private keys being logged to a server. The category of threat that has cost Solana users the most money is the one no token metadata scanner can see.

Phase 0 doesn't make integrity.molt a full formal verifier. We're not Certik or OtterSec. What we are is a practical first-pass tool that can run in seconds, catch the most common Rust and Solana-specific vulnerability classes, and — critically — use AI to separate real findings from noise so you're not buried in false positives.

The right mental model: Rugcheck tells you if the token looks safe. integrity.molt v2.1 tells you if the program looks safe. For most retail trading decisions, Rugcheck is fine. For anything involving meaningful capital or protocol integration, you want code analysis.
