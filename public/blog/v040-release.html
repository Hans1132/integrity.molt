LLM Advisor for Grey-Zone Risk Scores

Binary pass/fail labels break down exactly where they matter most: the messy middle. A token scoring 55 out of 100 isn't clean, but it isn't an obvious rug either — it might be a new legitimate project with thin liquidity, a fork with unreviewed code, or a genuinely suspicious contract hiding behind a low-profile launch. Until now, integrity.molt left that interpretation entirely up to you.

v0.4.0 introduces an LLM advisor module that activates automatically when a scan returns a risk score between 40 and 70. The module receives the structured signal set — authority flags, liquidity depth, holder concentration, freeze authority status, mint authority, metadata mutability — and returns a plain-language advisory explaining which specific signals are driving ambiguity and what to watch before committing capital.

Why 40–70? Scores below 40 carry clear red flags that speak for themselves. Scores above 70 are generally clean enough that added prose adds little. The 40–70 band is where nuance earns its keep, and where a well-prompted language model can surface context that a raw number cannot.

The advisor output is appended directly to the scan result card. It is clearly labelled as AI-generated guidance, not a financial recommendation. Think of it as a senior analyst's margin notes: useful context, not a verdict.

Server-Side Watchlist Enforcement (Critical Security Fix)

This is the most important security change in v0.4.0, and it deserves a direct explanation of what was wrong before.

Prior to this release, watchlist tier limits — the cap on how many tokens a Free, Basic, or Pro subscriber can monitor simultaneously — were enforced only in the UI. The API endpoints that add tokens to a watchlist accepted requests without independently verifying whether the authenticated user had remaining quota. A moderately technical user could bypass the frontend entirely with a direct curl call and add unlimited tokens regardless of their plan tier.

# What was previously possible on any tier:
curl -X POST https://intmolt.org/api/watchlist/add \
  -H "Authorization: Bearer <token>" \
  -d '{"mint": "So11111111111111111111111111111111111111112"}'

The server now performs an authoritative tier check on every watchlist mutation — add, bulk-import, and restore from backup. If the user is at or above their plan limit, the request returns a structured 403 with a clear upgrade prompt. The UI check remains in place as a UX convenience, but it is no longer the gatekeeper.

If you use the integrity.molt API directly: Watchlist POST endpoints now return 403 WATCHLIST_LIMIT_EXCEEDED when quota is exhausted. Update any automation that previously relied on the absence of server-side validation. See plan limits for current tier quotas.

Helius Webhook Filter: Eliminating AMM Noise

Solana's throughput is a double-edged sword for real-time monitoring. Watching a token account for any transaction — the previous ANY filter — means your webhook endpoint receives every swap, every micro-fee, every AMM rebalance tick. For a heavily traded token, that's hundreds of irrelevant events per minute obscuring the signals that actually matter for security monitoring: authority changes, large transfers to unlabelled wallets, mint events, and freeze operations.

v0.4.0 narrows the Helius webhook subscription to a targeted set of security-relevant transaction types only. The specific filter set covers:

  • Mint authority transfers and revocations
  • Freeze authority changes
  • Large token transfers above configurable thresholds
  • Program upgrade events on associated programs
  • Metadata mutation transactions

The practical result is a dramatic reduction in webhook volume for actively traded tokens — and a corresponding improvement in alert latency, since the processing queue is no longer backed up with AMM swap receipts. Security signals surface faster because they aren't competing with noise.

A rate limit of 300 requests per minute per webhook endpoint has also been applied at the Helius subscription layer, providing an additional backstop against traffic spikes during high-volatility periods.

Known-Safe Token Whitelist

Scanning SOL, USDC, or JitoSOL produces predictable results — they're established tokens with well-understood on-chain properties. Running them through the full scanner pipeline on every request wastes compute and adds latency for a result that isn't going to change.

The new known-safe token whitelist short-circuits the analysis pipeline for a curated set of high-confidence, widely-verified tokens. Scans against whitelisted mints return immediately with a confirmed-safe status and a provenance note explaining the basis for the classification. The ⚡ Instant result badge (also new in this release) appears on these results and on any scan served from cache, so you always know whether you're seeing a live analysis or a fast-path response.

The whitelist is maintained internally and is not user-editable. Additions follow a conservative review process — this is not a mechanism for token projects to self-certify.

38 Unit Tests for Scanner Logic

Scanner correctness is a product promise, not just an engineering nicety. If the logic that evaluates freeze authority status or scores holder concentration is wrong, every downstream decision — human or automated — is built on a broken foundation.

v0.4.0 ships 38 unit tests covering the core scanner logic, all designed to run without any network dependency. Tests use fixture data representing real-world token configurations: legitimate projects, known rugs, honeypots, tokens with revoked authorities, tokens with suspicious but not conclusive patterns.

The no-network constraint is intentional. Tests that depend on live RPC calls are slow, flaky, and can't be run in CI without managing credentials and rate limits. The fixture-based approach means the full test suite runs in under two seconds and is gated on every pull request merge.

This is the foundation for a public test report we plan to publish alongside future releases — so you can audit what we're actually testing, not just take our word for it.

UX Improvements: Cards, Tickers, and Soft Paywalls

Visual Scan Type Card Picker

The HTML <select> dropdown for choosing scan type has been replaced with a visual card picker. Each scan mode — token analysis, wallet risk, program audit — now has its own card with a short description of what it checks. The change reduces the "what does this mode actually do?" confusion that was the most common support question in the previous cycle.

Live Activity Ticker

The landing page now displays a live activity ticker showing scans run today, total scans processed, and current success rate. The stats refresh automatically every 60 seconds. It's a small thing, but real-time signals build more trust than static marketing copy — especially for a security tool where credibility is the product.

Soft Paywall Email Capture

Free-tier users who hit their scan quota now encounter a soft paywall before the hard block: a one-shot-per-session email capture that unlocks a small number of additional scans. This replaces an abrupt quota wall with a more gradual conversion path. The prompt appears once per session and is not repeated on subsequent visits. See plan options if you'd rather skip the prompt entirely with a paid tier.

FAQ Accordion & Feature Comparison Table

Eight frequently asked questions now live in an accordion section linked from the main navigation. Topics cover how scoring works, what data sources are used, how watchlist alerts are delivered, and what the difference between a token scan and a program audit is. Alongside the FAQ, a 15-feature comparison table across all four plan tiers replaces the previous abbreviated feature list — if you're evaluating whether to upgrade, the full picture is now one scroll away at intmolt.org/#plans.

X/Twitter Share Button & Friendly Error Messages

Scan results now include a one-click X/Twitter share button that pre-populates a tweet with the token name, risk score, and a link to run a fresh scan at intmolt.org/scan. Sharing a scan result is a lightweight way to flag suspicious tokens to your community without requiring anyone to trust your interpretation — they can verify independently in seconds.

Error handling has also been overhauled. Previously, quota exhaustion, invalid addresses, RPC timeouts, and unknown errors all surfaced as generic failure messages. Each error condition now returns a specific, actionable message: quota errors explain remaining limits and link to upgrade options; invalid address errors confirm the format expected; RPC timeouts include a retry prompt; unknown errors include a reference ID for support.

Upgrade & Compatibility Notes

  • API consumers: Handle the new 403 WATCHLIST_LIMIT_EXCEEDED response on watchlist mutation endpoints.
  • Webhook integrations: The narrowed Helius filter will reduce your incoming event volume significantly. If you were using webhook events to track all token activity rather than just security signals, you will need to supplement with a dedicated data stream.
  • Cached result badge: The instant_result: true field is now included in scan API responses for whitelisted tokens and cache hits. You can use this to distinguish live analysis from fast-path responses in your own tooling.

Ready to run a scan? Head to intmolt.org/scan — or review the full tier breakdown at intmolt.org/#plans if you're evaluating an upgrade.
