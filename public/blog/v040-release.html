UX overhaul: scanner (scan.html)

The most visible change is the scan type selector. The old HTML <select> dropdown is replaced with six visual radio-button cards — Quick, Token, EVM, Wallet, Pool, Deep. Each card shows the price, a one-line description, and highlights when selected. Clicking auto-detects the right scan type when you paste an address.

The paywall flow now has two stages. If you haven't given us your email this session, we show a soft "stay updated" prompt before the hard paywall. One-shot per browser session (stored in sessionStorage), never shown again once submitted, never shown for deep scan (which goes straight to payment). Captured emails go to the /email/capture endpoint and are logged for follow-up campaigns.

Error messages are now human. Instead of 429 or "scan failed", users see messages like "You've used all 3 free scans today. Come back tomorrow or upgrade for unlimited." — with a retry button where appropriate. The ERROR_MAP covers quota exhaustion, invalid address, RPC timeouts, and unknown errors.

Two small additions that improve trust signals: a cached results badge ("⚡ Instant result — cached scan") appears when a result is served from cache, so users know they're not hitting the network. And a X/Twitter share button lets users post their risk score with a pre-written tweet.

Live activity ticker

A slim green bar below the header now shows real-time stats: scans today, total scans, and success rate — pulled from /api/v2/stats and auto-refreshed every 60 seconds. On load failure it falls back to a static message. The pulsing dot keeps it visually alive without being distracting.

FAQ section

Eight collapsible questions before the footer cover the things we kept getting asked in Telegram: Is it really free? What's the difference between quick scan and Deep Audit? What is x402? What does Ed25519 signed mean? Accordion-style: opening one closes the others. Linked from header nav as #faq.

Feature comparison table

A full-width matrix below the pricing cards compares all four plans across 15 features — scan types, watchlist limits, alert channels, delta reports, API access, SLA. The Builder column is highlighted in green. On mobile it scrolls horizontally.

Watchlist tier limits

Free users have always been limited to 3 watchlist addresses by design, but that limit wasn't enforced in code — it was only shown in the UI. v0.4.0 enforces it server-side in both API paths (Telegram bot and REST API).

// server.js — enforced limits
const WATCHLIST_TIER_LIMITS = {
  free:        3,
  pro_trader: 20,
  builder:   100,
  team:      500,
};

// Adding a 4th address on free tier returns:
HTTP 403 {
  "error": "Watchlist limit reached (3/3). Upgrade to Pro Trader for 20 addresses.",
  "tier": "free",
  "limit": 3,
  "current": 3
}

The check runs before the Helius webhook sync — so adding an address over the limit neither saves to the DB nor registers with Helius. Users get a clear upgrade message with their current tier and the next tier's limit.

Helius webhook hardening

This was the riskiest silent issue before this release. The previous webhook configuration used transactionTypes: ["ANY"] — meaning Helius would push every transaction for every watched address. For a quiet wallet this is fine. For an address associated with a Raydium pool or active DeFi protocol, this can mean tens of thousands of webhooks per hour.

v0.4.0 replaces ANY with a targeted list of security-relevant transaction types only:

const SECURITY_TX_TYPES = [
  'SET_AUTHORITY',
  'UPGRADE_PROGRAM_INSTRUCTION',
  'CLOSE_ACCOUNT',
  'BURN',
  'BURN_NFT',
  'TRANSFER',
  'TRANSFER_CHECKED',
  'MINT_TO',
  'INITIALIZE_MINT',
];

This eliminates swap transactions, AMM rebalances, liquidity events — the high-volume noise that drives Helius credit consumption without adding security signal. The filter applies to all four Helius API calls: setupWebhook, addAddresses, removeAddresses, and syncWatchlistToWebhook.

Two additional safeguards:

  • Global rate limit: max 300 webhook requests per minute at the Express layer. Requests over the limit get a 200 OK with { ok: false, error: "rate limited" } — returning 200 prevents Helius retry storms.
  • Disk cap: events.jsonl stops growing at 50 MB. The check runs every 100 webhook calls (not every call) to avoid stat() overhead. When paused, events are silently dropped rather than crashing the process.

Unit tests

38 tests covering pure scanner logic — no network, no DB, no RPC calls required. Run with:

node tests/scan-logic.test.js

Coverage: scoreToCategory boundary conditions, analyzeConcentration thresholds, WEIGHTS sum validation, analyzeSource pattern matching, detectFees, analyzeTransfers, ABI decoders (decodeString, decodeUint256, decodeAddress), and cross-scanner WEIGHTS consistency check.
