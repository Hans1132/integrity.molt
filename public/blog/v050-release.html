Scam Token Database: Check First, Scan Second

One of the most impactful additions in v0.5.0 is also one of the most straightforward: before the scanner fires up its LLM pipeline, it now cross-references the submitted token against a unified archive of 17,000+ known scam tokens sourced from SolRPDS and RugCheck. If there's a match, the verdict is returned immediately — no GPU time wasted, no latency, no chance of the model talking itself into a charitable interpretation of a confirmed rugpull.

This matters for two reasons. First, it dramatically reduces false negatives on tokens that have already been catalogued by the broader security community. Second, it lowers cost-per-scan by short-circuiting the expensive path for already-known bad actors. The database is refreshed continuously and the lookup happens in-process at scan time, adding negligible overhead.

Why this order matters: Running deterministic database lookups before probabilistic LLM analysis means known scams are caught with 100% consistency, and the model's attention is reserved for genuinely novel or ambiguous cases where it can add real value.

EVM Scam Detection: Beyond Solana

integrity.molt began as a Solana-native tool, and that remains its primary focus. But the reality of DeFi in 2025 is that bad actors deploy the same scam blueprints across every EVM chain with minimal modification. v0.5.0 extends detection coverage to Ethereum, BSC, Polygon, Arbitrum, and Base with three core analysis modules:

  • Fee asymmetry analysis — detects tokens with asymmetric buy/sell tax configurations commonly used to trap holders (e.g., 1% buy tax, 49% sell tax).
  • Holder concentration check — flags tokens where a small number of wallets control an outsized share of supply, a leading indicator of coordinated dump risk.
  • Mint/burn risk patterns — identifies contracts with unrestricted or owner-controlled mint functions that can inflate supply post-launch.

EVM token audits are available at the same unified price point as Solana scans ($0.75 per scan — see pricing changes below). If you're evaluating a cross-chain project or a Solana token that also has a wrapped EVM presence, you can now audit both sides of that exposure in one workflow at intmolt.org/scan.

LLM Scan Validator: Closing the Hallucination Gap

LLM-based security analysis introduces a class of failure modes that don't exist in deterministic rule engines: the model can hallucinate a clean verdict for a clearly malicious token, or produce a risk score that contradicts the evidence it just cited. v0.5.0 ships a post-generation validation layer that sits between the raw LLM output and the response returned to the user.

The 6 validation rules

  • Schema enforcement — output must conform to a strict JSON schema; malformed responses are rejected and re-queried.
  • Score-evidence consistency — a low risk score paired with high-severity findings triggers a validation failure.
  • Prompt anchoring — the validator checks that conclusions reference the actual on-chain data passed in, not fabricated addresses or metrics.
  • Null field rejection — required fields (risk score, summary, flag list) cannot be null or empty.
  • Severity floor enforcement — if any critical flag is present, the overall risk level cannot be "low" or "informational".
  • Contradiction detection — contradictory statements within the same report (e.g., "no mint authority" and "mint authority risk present") are flagged and the scan is retried.
False negatives are the dangerous failure mode. A false positive (flagging a clean token) wastes a user's time. A false negative (clearing a scam token) can cost real money. The validator is weighted to be conservative: when in doubt, it escalates rather than clears.

Golden Dataset Regression Suite

Shipping an LLM validator is only useful if you can verify it keeps working across model updates, prompt changes, and dependency upgrades. The new regression suite locks in a golden dataset of 17 tokens (drawn from SolRPDS and Jupiter listings) with ground-truth verdicts, and runs 29 accuracy tests against them on every deployment via test-gate.sh.

# Run the accuracy gate locally before deploying
./test-gate.sh --dataset golden --threshold 0.95

# Output example:
# [PASS] 27/29 tests passed (93.1%)
# [FAIL] Threshold 95.0% not met — deployment blocked

The gate is configurable and the threshold can be adjusted per environment, but the default blocks deployment if accuracy drops below 95% on the golden set. This gives the team a hard stop against regressions introduced by model version changes or prompt refactors — a category of bug that's notoriously hard to catch with conventional unit tests.

Scan accuracy signals are now also persisted to the scan_accuracy_signals table in real time, and aggregated metrics are exposed via the /api/v1/admin/accuracy endpoint. Users can submit feedback on scan results via /api/v1/feedback, which feeds back into the accuracy monitoring pipeline.

Infrastructure & Bug Fixes

Payment middleware hardening

Two significant payment bugs are resolved in this release. SOL and USDC payment thresholds were previously sharing a single comparison path, meaning a SOL payment could be incorrectly validated against USDC thresholds and vice versa — these are now fully separated. Additionally, ATA (Associated Token Account) verification and anti-replay protection have been added to the payment middleware to prevent double-spend and replay attack vectors on on-chain payments.

Helius RPC deduplication

The Helius RPC client in fork.js was firing redundant calls for the same account data within a single scan session. Deduplication now caches responses within the scan context, reducing unnecessary RPC consumption and improving scan latency on complex wallets with many token accounts.

CAPTCHA replacement

Cloudflare Turnstile has been replaced with an HMAC-signed server-side math CAPTCHA. This eliminates the external dependency on Cloudflare's challenge infrastructure, which was causing occasional failures in environments where Cloudflare's JS challenge couldn't complete (certain VPN configurations, aggressive privacy browsers). The new CAPTCHA has no external dependency and is validated entirely server-side.

SQLite WAL and event log management

Two separate unbounded growth issues are addressed. The events.jsonl audit log now has a hard 50 MB cap with automatic rotation — previously it would grow indefinitely on busy instances. SQLite WAL checkpointing now runs every 6 hours, preventing the write-ahead log from accumulating across long-running server sessions. Both changes are important for self-hosted deployments where disk space is finite.

Other fixes

  • scans_today counter — was always returning zero due to a date format mismatch between the application layer and the SQL DATE() function. Fixed by normalizing to ISO 8601 throughout.
  • Google OAuth — the OAuth strategy was not being registered on server startup under certain initialization orders. Now explicitly registered during the auth middleware setup phase.
  • Scan type card UX — the entire card is now a click target, not just the button within it. A small change that meaningfully improves the scan initiation flow on mobile.

Getting started with v0.5.0

Existing users on pay-per-scan will see no change to their workflow — all standard scans remain at $0.75 and the interface is backward compatible. If you're running more than ~20 scans per month or need continuous watchlist monitoring, the Pro Trader tier will be more economical and gives you alerting that point-in-time scanning can't provide.

To audit a Solana or EVM token, wallet, or liquidity pool, head to intmolt.org/scan. EVM chain selection is available in the scan type dropdown when a non-Solana address format is detected.

Self-hosted users: Pull the v0.5.0 tag and run ./test-gate.sh before deploying to production. The new WAL checkpoint and event log rotation settings are active by default — no configuration required, but review config/storage.js if you need to adjust thresholds for your disk constraints.
