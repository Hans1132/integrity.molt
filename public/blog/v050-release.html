Scam Token Database: catch known bad actors before the LLM even wakes up

The most expensive part of a scan — both in latency and cost — is the large language model inference step. In v0.5.0, integrity.molt now pre-filters every token address against a merged archive of 17,000+ confirmed scam tokens sourced from SolRPDS and RugCheck before any LLM invocation occurs. If a token is already known-bad, you get an instant, high-confidence verdict with zero AI inference overhead.

This matters for three reasons:

  • Speed: Known scam tokens return results in milliseconds rather than seconds.
  • Cost efficiency: Skipping LLM calls for already-catalogued rugs reduces infrastructure cost, which is why we can keep standard scans at $0.75.
  • Coverage: Many scam tokens share deployer wallets, bytecode patterns, or metadata strings. The database captures historical signal that on-chain heuristics alone might miss for a freshly relaunched variant.
The scam database is consulted at scan time, not just at index time. Every scan you run against a Solana token address now benefits from this lookup automatically — no configuration required.

EVM Scam Detection: the same rigor, now across five chains

Solana's speed makes it a preferred playground for rug pulls, but EVM chains have their own vocabulary of scam patterns. v0.5.0 extends integrity.molt's detection engine to Ethereum, BSC, Polygon, Arbitrum, and Base with three dedicated analysis modules:

Fee asymmetry analysis

Many EVM honeypot contracts implement asymmetric transfer taxes — a low buy fee to attract liquidity and an arbitrarily high (sometimes 100%) sell fee to trap holders. The scanner now reads the token contract's fee logic and flags any buy/sell fee disparity above configurable thresholds.

Holder concentration check

A token where the top 5 wallets hold more than 60% of supply (excluding locked liquidity) is a significant rug risk. The checker fetches on-chain holder distribution and surfaces concentration scores directly in the scan report.

Mint and burn risk patterns

Uncapped mint authority and unrestricted burn functions on the owner address are classic pre-rug indicators. The EVM module inspects contract bytecode and ABI for these patterns, even when source code is not verified on the target block explorer.

LLM scan validator: making AI output trustworthy by construction

LLM-powered security analysis introduces a class of failure that traditional rule-based scanners don't have: the model can hallucinate a detail, omit a critical risk, or produce output that doesn't conform to the expected schema. Previous versions of integrity.molt relied on prompt engineering alone to prevent this. v0.5.0 introduces a dedicated LLM scan validator as a post-inference enforcement layer.

Six validation rules

  • Schema conformance: Output is validated against a strict JSON schema before being stored or returned.
  • Risk field completeness: Every identified risk must carry an id, severity, and detail — partial objects are rejected.
  • Severity range enforcement: Severity values must be one of LOW, MEDIUM, HIGH, CRITICAL — free-text severities are rejected.
  • Prompt anchoring: The validator checks that the model's output references the actual token address from the prompt, not a hallucinated address.
  • False negative suppression: If the database pre-filter flagged the token as a known scam but the LLM returned a clean verdict, the validator overrides the LLM and surfaces the database finding.
  • Confidence floor: Scan results below a minimum confidence threshold are marked as INCONCLUSIVE rather than silently passed through as clean.

When a scan output fails validation, the system retries the inference with a refined prompt rather than returning a potentially misleading result. This increases average scan latency by a small margin in edge cases, but eliminates a category of silent failure that could lead traders to trust a compromised token.

Golden dataset regression suite: accuracy as a deployment gate

Shipping a new LLM prompt, updating detection heuristics, or integrating a new data source all carry the risk of regressing scan accuracy in subtle ways. The new golden dataset regression suite makes accuracy a hard gate in the deployment pipeline.

  • 17 reference tokens drawn from SolRPDS and Jupiter with known ground-truth verdicts (confirmed scam, confirmed safe, or confirmed high-risk)
  • 29 accuracy tests covering risk identification, severity assignment, and report field completeness
  • test-gate.sh — a CI script that runs the full suite and blocks deployment if the pass rate drops below threshold
# test-gate.sh excerpt
PASS_THRESHOLD=27  # out of 29 tests
RESULT=$(node run-accuracy-suite.js | tail -1)
if [ "$RESULT" -lt "$PASS_THRESHOLD" ]; then
  echo "Accuracy gate failed: $RESULT/29 — deployment blocked"
  exit 1
fi

This is a meaningful shift in how integrity.molt is developed: accuracy is no longer validated manually on an ad-hoc basis before releases. Every merge that touches the scan pipeline must pass the golden dataset gate.

Security and infrastructure fixes

Payment middleware hardening

Two significant payment bugs were resolved in this release. First, SOL and USDC payment thresholds were previously stored in the same comparison path, meaning a USDC payment could be incorrectly validated against a SOL price threshold (and vice versa). These are now separated with explicit currency guards. Second, Associated Token Account (ATA) verification and anti-replay protection were added to the payment middleware — preventing edge cases where a transaction signature could be submitted more than once.

CAPTCHA: removing the external dependency

Cloudflare Turnstile has been replaced with an HMAC-signed math CAPTCHA. The challenge is generated server-side, signed with a secret key, and verified on submission — no third-party JavaScript, no external network call, no privacy tradeoff. For a security-focused tool, eliminating an opaque external dependency from the authentication flow is the right call.

Helius RPC deduplication

A bug in fork.js caused redundant on-chain calls to be made to the Helius RPC endpoint during certain scan paths. Beyond the unnecessary cost, duplicate calls introduced a small race condition window where two in-flight requests could return slightly inconsistent account state. Deduplication logic now consolidates identical requests within a scan cycle.

WAL and event log caps

Two infrastructure hygiene fixes prevent unbounded disk growth in long-running deployments:

  • events.jsonl is now capped at 50 MB with automatic log rotation.
  • SQLite WAL checkpoints now run on a 6-hour schedule, preventing the write-ahead log from growing indefinitely between restarts.

scans_today fix

A date format mismatch in the SQL query for scans_today caused the counter to always return zero. The column was stored as YYYY-MM-DD but the query was comparing against an ISO timestamp string. This affected dashboard display and rate-limit enforcement logic — both now use consistent date formatting.

Upgrade notes

If you are running a self-hosted instance, apply the database migration for scan_accuracy_signals before starting the v0.5.0 server. The /api/v1/feedback and /api/v1/admin/accuracy endpoints will return 500 if the table does not exist. The WAL checkpoint job starts automatically on process boot — no cron configuration is required.
